# Автоматичне списання/повернення товару на складі — налаштування

Код готовий і протестований. Лишився один ручний крок у Supabase — виконати
SQL-скрипт (агент не має прямого доступу до бази цього проєкту, лише до
REST/даних, тож DDL/тригери можна створити лише через SQL Editor вручну).

Це важливий крок: у вашому проєкті немає вбудованого помічника "Database
Webhooks" у формі створення тригера (немає готової функції `http_request` у
списку "Function to trigger") — тож замість UI-майстра робимо це напряму
через розширення `pg_net`. Це той самий механізм, яким Supabase реалізує
свої Webhooks під капотом — результат ідентичний.

## Крок 1 — виконати SQL

1. Supabase Dashboard → ваш проєкт → **SQL Editor → New query**.
2. Відкрити [`setup-inventory-sync.sql`](./setup-inventory-sync.sql).
3. **Перед запуском замінити** в тексті скрипта (2 функції, по 2 місця кожна —
   4 заміни разом):
   - `<YOUR_WEBHOOK_URL>` → `https://zipper-new-shop.vercel.app/api/webhooks/inventory-sync`
   - `<YOUR_WEBHOOK_SECRET>` → значення `INVENTORY_WEBHOOK_SECRET` з `.env`
4. Вставити відредагований текст у SQL Editor і натиснути **Run**.

Скрипт зробить одразу все:
- додасть до `orders_returns` поля `oid`, `product`, `warehouse_id`, `qty`,
  `restocked` (старі записи не постраждають — усі нові поля nullable);
- увімкне розширення `pg_net`;
- створить 2 функції + 2 тригери на `orders_item` (insert/update/delete) і
  `orders` (update), які надсилають подію на `/api/webhooks/inventory-sync`.

## Крок 2 — задеплоїти код

Усе, що написано сьогодні (сам вебхук-роут, форма повернень, фікс
`proxy.ts`), поки що лише локально — на `zipper-new-shop.vercel.app` цього
коду ще немає, доки не буде коміт + push (і Vercel не задеплоїть).

## Крок 3 — секрет у Vercel

`.env` на проді не використовується — потрібно окремо додати
`INVENTORY_WEBHOOK_SECRET` (те саме значення) у Vercel Dashboard → проєкт →
**Settings → Environment Variables** (Production), і зробити redeploy.

## Що це дає

- **Товар додано в замовлення** (нове замовлення з магазину / вручну через
  адмінку) → одразу списується зі складу (може піти в мінус — це навмисно,
  поки реальні залишки не введені).
- **Кількість у рядку замовлення змінено** → попередня кількість повертається,
  нова списується.
- **Рядок/замовлення видалено** → кількість повертається на склад.
- **Замовлення скасовано** (статус → «Скасовано») → весь товар замовлення
  повертається на склад; якщо скасування скасувати назад — списується знову.
- **Оформлено повернення** (нова форма на сторінці замовлення) → товар одразу
  повертається на склад.

Склад для цих операцій обирається автоматично — активний склад із
найменшим `priority` (зараз це «Зоопарк»).

## Як перевірити, що спрацювало

Після виконання SQL і деплою — відкрийте будь-яке замовлення, зайдіть у
режим редагування товарів, змініть кількість і збережіть. Потім перевірте
залишок цього товару на сторінці **Залишки на складі** — число має
змінитись. Якщо ні — перевірте у Supabase **Database → Triggers**, що обидва
тригери (`trg_inventory_sync_items`, `trg_inventory_sync_orders`) активні, і
що URL/секрет у функціях відповідають реальним значенням.
